# Webrivio Workflow

The reusable Webrivio starter is configured for visual-first, scroll-animated, image-aware client builds. This file documents the end-to-end workflow.

## New client workflow

1. Copy the starter into a new client folder using `client-sites\new-site.ps1`.
2. Open the new project and fill out `CLIENT_INFO.md` completely. Required fields halt the build if empty.
3. If real client photos are available, drop them into `/public/uploads`. Use the suggested filenames in CLIENT_INFO.md section 7 when possible.
4. If real photos are not available, make sure `PEXELS_API_KEY` is reachable through one of:
   - `.env.local` in the new client project
   - the `PEXELS_API_KEY` environment variable
   - `.env.local` in the parent `webrivio-starter` folder (used as automatic fallback)
5. Run `npm install`.
6. Run `npm run fetch-assets` if you need Pexels placeholders. Use `npm run fetch-assets:force` to overwrite existing placeholder files.
7. Start the Claude Code session: `claude`.
8. Run `/webrivio-build` inside Claude Code.
9. Preview locally with `npm run dev`.
10. Polish once if needed.
11. Final check with `npm run build`.

## Asset folder convention

- `/public/uploads` — **real client images.** Logos, photos, videos, anything provided by the client.
- `/public/placeholders` — **temporary Pexels placeholder images.** Demo-only atmosphere.
- `/public/inspiration` — screenshots, reference notes, and design inspiration. Not used at runtime.

Placeholder images are **not** client proof. They must never be presented as the client's real work, real team, real customers, real projects, real before/after results, or real testimonials.

`ASSET_CREDITS.md` is generated by `npm run fetch-assets` and tracks every placeholder source (file name, query, photographer, Pexels URL).

Real images should replace placeholders before final delivery whenever possible.

## Environment & secrets

- Never commit `.env.local`. It is git-ignored by default.
- `.env.example` documents which keys are expected.
- The asset fetcher reads `PEXELS_API_KEY` from `.env.local`, the environment, or the starter fallback path. It never prints the key.

## Skills

- `/webrivio-build` is the main project-local skill. It orchestrates the full visual-first build from `CLIENT_INFO.md`.
- `frontend-design` is included as a **project-local** helper skill to improve design quality (typography, color, rhythm, motion, image treatment). It is intentionally project-specific so the workflow does not depend on global plugins.

Both skills live under `.claude/skills/` and travel with the starter when copied.

## Visual standard

See `docs/WEBRIVIO_VISUAL_SYSTEM.md` for the full visual-first standard: section anchors, copy limits, scroll-animation requirements, asset priority, mobile rules, and the calibration bar that makes the $497 offer feel like an obvious yes.

## Commands

| Command | Purpose |
|---------|---------|
| `npm install` | Install dependencies. |
| `npm run dev` | Start the Next.js dev server. |
| `npm run build` | Final build check. |
| `npm run lint` | ESLint. |
| `npm run fetch-assets` | Pull Pexels placeholders into `/public/placeholders` and write `ASSET_CREDITS.md`. Skips existing files. |
| `npm run fetch-assets:force` | Same as above, overwriting existing placeholder files. |

## Reminders

- `/webrivio-build` remains the main command for any client build.
- Real assets always beat placeholders.
- Placeholders are never proof.
- Never invent reviews, awards, certifications, licenses, years in business, guarantees, prices, testimonials, before/after results, or project outcomes.
- The first draft should already feel client-presentable, not a wireframe.
